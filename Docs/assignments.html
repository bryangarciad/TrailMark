Assignments

Nine graded builds. Each is the in-class build expressed as acceptance criteria, so no extra work is created outside the session. Class 4 of each course is an integration / capstone day with no separate deliverable. Copy any card straight into Canvas.

Standard submission (applies to every assignment) Unless noted, submit three things in Canvas:
  • Screen recording (file upload, .mp4/.mov): 1–3 min demonstrating the required behavior, including the empty/denied states where relevant. Device recording required for sensor features (2.3, 3.2).
  • Source (website URL): a link to your Git repo at the tagged commit, or a zipped project upload if no repo.
  • Reflection (text entry): 150–300 words answering the assignment's "explain" prompt.
Recommended Canvas setup: Online submission with File Uploads + Website URL + Text Entry enabled, 100 points, due end of session +1 week.

Assignment 1.1 100 pts iOS

"Today" dashboard

Build spec. On launch, request HealthKit authorization. Display today's steps, distance, and active energy, refreshing when the view appears. Handle denied/undetermined states gracefully with a clear empty state and no crashes. All HealthKit calls live in TrailmarkCore, never in the view.

Acceptance criteria

  • App requests Health authorization on first launch with clear purpose strings.
  • Today's steps, distance, and active energy display and refresh on .onAppear / pull-to-refresh.
  • Denied / no-data states show a clear empty state — no crash, no spinner-forever.
  • No HealthKit type appears in any View file; all queries are in the package.
  • Uses statistics queries (not raw sample enumeration) for the totals.

Expected learning outcomes

  • Set up an Xcode project with an app target plus a shared local Swift package, and explain why a shared module prevents duplication later.
  • Explain HealthKit's authorization and privacy model, and request read access correctly with clear purpose strings.
  • Read step count, distance, and active energy using statistics queries.
  • Render today's activity summary in SwiftUI while keeping health logic inside the core package.

Reflection prompt

Why does iOS never tell you whether read access was granted, and how did you design the UI around that?

Rubric

CriterionPts
Reads & displays all three metrics correctly40
All HealthKit logic in the shared package (architecture)25
Graceful authorization / empty-state handling15
Reflection demonstrates understanding of the privacy model10
Submission complete (recording + source + reflection)10
Assignment 1.2 100 pts iOS

"Field journal"

Build spec. Record a voice memo and a short video memo. Save each with metadata into the TrailmarkCore media store. Show a list with thumbnails and duration, and a detail view that plays the item back. Deleting a memo also removes its file from disk.

Acceptance criteria

  • Records audio (and video where the camera/library is available) and saves it with metadata: id, type, date, duration.
  • List shows each memo with a thumbnail (video) or icon (audio) and its duration.
  • Detail view plays the memo back.
  • Deleting a memo removes both the index entry and the file on disk (no orphaned bytes).
  • Media model + store live in the package; file names are stored relative, not as absolute URLs.

Expected learning outcomes

  • Capture video and audio with AVFoundation, and reason about capture sessions and permissions.
  • Persist media to app storage and model its metadata.
  • Generate video thumbnails and play media back in-app.
  • Build a memo list and detail screen backed by the shared media store.

Reflection prompt

Why store a relative file name instead of an absolute URL, and what breaks if you don't?

Rubric

CriterionPts
Capture + save with correct metadata30
List (thumbnail + duration) and working playback30
Delete removes the file from disk15
Store lives in the package; sound persistence design15
Submission complete10
Assignment 1.3 100 pts iOS

"Recovery view"

Build spec. Save a sample activity as a workout and confirm it appears in Health. Pull last night's sleep and the last seven days of active energy. Show a recovery screen: last night's sleep duration plus a 7-day energy chart. All queries live in TrailmarkCore.

Acceptance criteria

  • Saves a sample activity as an HKWorkout; student demonstrates it appearing in the Health app.
  • Reads last night's sleep via the sleep-analysis category type and shows a duration.
  • Computes the last 7 days of active energy and renders it with Swift Charts.
  • All reads/writes are in the package.

Expected learning outcomes

  • Write a completed activity to HealthKit as an HKWorkout and verify it lands in the Health app.
  • Query sleep analysis for last night using a category type.
  • Compute simple insights such as sleep duration and a weekly active-energy trend.
  • Visualize those insights with Swift Charts.

Reflection prompt

What date window did you use to capture "last night's" sleep, and why is bracketing it correctly harder than it looks?

Assignment 2.1 100 pts watchOS

"Wrist home"

Build spec. Add the watch target and consume TrailmarkCore with no duplicated model code. Build a glanceable home screen showing today's headline metric and one quick action, readable in about two seconds. Justify one design choice against a specific watchOS guideline.

Acceptance criteria

  • Watch target links the shared package — zero copied model/manager code.
  • Home screen shows one headline metric + one quick action, glanceable in ~2 seconds.
  • Layout is wrist-appropriate (not a shrunk iPhone screen).
  • Reflection names a specific watchOS guideline and justifies a choice against it.

Expected learning outcomes

  • Summarize Apple's wearable design principles: glanceability, focused tasks, hierarchy, Digital Crown ergonomics, and when to defer to the phone.
  • Add a watchOS target and import TrailmarkCore — reuse models and managers instead of rewriting them.
  • Translate an iOS screen into a wrist-appropriate layout.
  • Build a glanceable watch home screen.

Reflection prompt

Pick one design choice on your home screen and defend it against a named watchOS Human Interface guideline.

Rubric

CriterionPts
Watch target reuses the shared core (no duplication)30
Glanceable headline metric + one quick action30
Wrist-appropriate layout & hierarchy20
Reflection cites a specific guideline10
Submission complete10
Assignment 2.2 100 pts watchOS

"Wrist memo"

Build spec. On the watch, record a voice memo, save it via the shared media model, list saved memos, and play one back. Keep the interface to the essentials a wrist actually needs.

Acceptance criteria

  • Records audio on the watch and saves via the same MediaStore / model used on iOS.
  • Lists saved memos and plays one back.
  • Interface is pared to wrist essentials (no iPhone-style density).
  • No duplicated media model — the watch reuses the package.

Expected learning outcomes

  • Record and play audio on watchOS and reason about watch-specific media constraints: storage, audio session, and limited UI.
  • Reuse the media model from TrailmarkCore while accepting a watch-appropriate interface.
  • Contrast the iOS and watchOS capture experience and explain why they differ.

Reflection prompt

What did you deliberately leave OUT of the watch capture UI compared to iOS, and why?

Rubric

CriterionPts
Record + save via shared model35
List + playback works on the watch30
Wrist-appropriate, essentials-only UI15
Reflection contrasts iOS vs watchOS capture10

Assignment 3.1 100 pts iOS + watchOS

"Pocket sync"

Build spec. Record an activity and a voice memo on the watch, then sync both to the iPhone over WatchConnectivity so they appear in the iOS journey list. Choose the right transfer type for each payload and explain why.

Acceptance criteria

  • WCSession established and activated on both sides.
  • An activity recorded on the watch appears in the iOS list after sync.
  • A voice-memo file transfers from watch to phone and is playable on iOS.
  • Reflection justifies the transfer type chosen for each payload (message vs userInfo vs applicationContext vs file).

Expected learning outcomes

  • Explain the WCSession transfer types — live messages, application context, userInfo, and file transfer — and when each fits.
  • Establish a session on both sides and sync a recorded activity from watch to phone.
  • Transfer a media file such as a voice memo across devices.

Reflection prompt

For each payload (summary, activity record, memo file) name the transfer type you chose and why the others were wrong.

Assignment 3.2 100 pts watchOS · device required

"Go for a walk"

Build spec. Build a live activity that starts on the watch, keeps updating heart rate, elapsed time, and energy while backgrounded, and saves a real HKWorkout on finish. Confirm it lands in the Health app.

Device required Workout sessions + background runtime need a physical Apple Watch; record the demo on device.

Acceptance criteria

  • Heart rate, elapsed time, and energy update — including while the app is backgrounded (Workout Processing mode enabled).

Expected learning outcomes

  • Start an HKWorkoutSession and stream metrics with a live workout builder.
  • Keep the app running and updating in the background during an activity.
  • Save the finished workout to HealthKit.

Assignment 3.3 100 pts watchOS

"Make it last"

Build spec. Profile one feature — for example live motion plus UI refresh plus sync. Make at least two optimizations and document the before/after measurements with a short note on the tradeoff you chose.

Acceptance criteria

  • Profiles a real feature with Instruments (Energy / Time Profiler / Allocations).
  • Makes at least two optimizations (e.g. sampling rate, UI refresh cadence, batched sync).
  • Documents before/after numbers — not impressions.
  • Explains the tradeoff each optimization made.

Expected learning outcomes

  • Profile CPU, energy, and memory on watchOS with Instruments.
  • Tune sensor sampling rates and UI refresh to cut battery cost.
  • Reduce connectivity overhead by choosing batched over chatty sync.

Submission note

Replace the screen recording with a short written report (PDF or text) containing before/after Instruments screenshots and the two optimizations. Source link still required.

Rubric

CriterionPts
Valid Instruments profiling of a real feature25
Two genuine optimizations30
Before/after measurements documented25
Tradeoff analysis10
Submission complete10
