iOS & watchOS · One project · 12 sessions

Trailmark.

A single adventure-journal app, built across three courses. It records where you went, how you moved, how your body responded, and what you captured along the way — synced between wrist and pocket.

The architecture that kills the redundancy

Course 1 builds the iPhone app and a shared Swift package, TrailmarkCore — models, the HealthKit manager, the route engine, and the media store. Course 2 adds a watchOS target that simply imports it, so we never rewrite the same logic for two platforms. "Don't repeat yourself across devices" becomes a lesson, not a problem.

01

Course One

iOS — Foundations & the Shared Core

Build the iPhone app and the reusable TrailmarkCore package. Everything later in the program stands on this layer.

Class 1.1 · Session 01 of 12

Architecture & the HealthKit foundation

SwiftUIHealthKitSwift PackageHKStatisticsQuery
Learning outcomes
  • Set up an Xcode workspace with an app target plus a shared local Swift package (TrailmarkCore), and explain why a shared module prevents duplication later.
  • Explain HealthKit's authorization and privacy model, and request read access correctly with clear purpose strings.
  • Read step count, walking/running distance, and active energy using statistics queries.
  • Render today's activity summary in SwiftUI, keeping all health logic inside the core package.
Assignment · build spec

"Today" dashboard

On launch, request HealthKit authorization. Display today's steps, distance, and active energy, refreshing when the view appears. Handle denied/undetermined states gracefully with a clear empty state and no crashes. All HealthKit calls live in TrailmarkCore, never in the view.

Class 1.2 · Session 02 of 12

Capturing the moment — media management

AVFoundationAVAudioRecorderFileManagerThumbnails
Learning outcomes
  • Capture video and audio with AVFoundation, and reason about capture sessions and permissions.
  • Persist media to app storage and model its metadata: id, type, date, duration, optional coordinates.
  • Generate video thumbnails and play media back in-app.
  • Build a memo list and detail screen backed by the shared media store.
Assignment · build spec

"Field journal"

Record a voice memo and a short video memo. Save each with metadata into the TrailmarkCore media store. Show a list with thumbnails and duration, and a detail view that plays the item back. Deleting a memo also removes its file from disk.

Class 1.3 · Session 03 of 12

Two-way HealthKit — workouts in, sleep & recovery out

HealthKit (write)HKWorkoutsleepAnalysisSwift Charts
Learning outcomes
  • Write a completed activity to HealthKit as an HKWorkout and verify it lands in the Health app.
  • Query sleep analysis for last night using a category type.
  • Compute simple insights such as sleep duration and a weekly active-energy trend.
  • Visualize those insights with Swift Charts.
Assignment · build spec

"Recovery view"

Save a sample activity as a workout and confirm it appears in Health. Pull last night's sleep and the last seven days of active energy. Show a recovery screen: last night's sleep duration plus a 7-day energy chart. All queries live in TrailmarkCore.

Class 1.4 · Session 04 of 12

Where it happened — location & the unified journey

CoreLocationMapKitPolylineGeotagging
Learning outcomes
  • Request when-in-use location authorization and record a coordinate track.
  • Geotag media memos with the location where they were captured.
  • Render a route polyline with memo pins on a MapKit map.
  • Assemble a "Journey detail" screen uniting route, health, and media — the iOS app is now whole and ready for the wrist.
Integration day — no assignment. We connect the three previous builds into one screen and prepare the codebase to be consumed by a watch target.
02

Course Two

watchOS — Designing & Building for the Wrist

Reuse TrailmarkCore on a new watch target. Spend time only on what's genuinely different: wrist-first design and on-device sensing.

Class 2.1 · Session 05 of 12

Design for wearables + reusing the core

watchOS targetSwiftUI (watch)Digital CrownTrailmarkCore reuse
Learning outcomes
  • Summarize Apple's wearable design principles: glanceability, focused tasks, hierarchy, Digital Crown ergonomics, and when to defer to the phone.
  • Add a watchOS target and import TrailmarkCore — reuse models and managers instead of rewriting them.
  • Translate an iOS screen into a wrist-appropriate layout.
  • Build a glanceable watch home screen.
Assignment · build spec

"Wrist home"

Add the watch target and consume TrailmarkCore with no duplicated model code. Build a glanceable home screen showing today's headline metric and one quick action, readable in about two seconds. Justify one design choice against a specific watchOS guideline.

Class 2.2 · Session 06 of 12

Voice memos on the wrist — media under constraints

AVFoundation (watch)AVAudioSessionLocal storage
Learning outcomes
  • Record and play audio on watchOS and reason about watch-specific media constraints: storage, audio session, and limited UI.
  • Reuse the media model from TrailmarkCore while accepting a watch-appropriate interface.
  • Contrast the iOS and watchOS capture experience and explain why they differ.
Assignment · build spec

"Wrist memo"

On the watch, record a voice memo, save it via the shared media model, list saved memos, and play one back. Keep the interface to the essentials a wrist actually needs.

Class 2.3 · Session 07 of 12

HealthKit on the watch

HealthKit (watch)Live metricsShared manager
Learning outcomes
  • Request HealthKit authorization on watchOS and explain how the watch's sensor access differs from the phone's.
  • Read current health metrics — heart rate, steps, active energy — on the watch using the shared HealthKit layer.
  • Present live-updating metrics in a wrist-appropriate layout.
Assignment · build spec

"Live vitals"

On the watch, after authorization, show current heart rate, steps, and active energy, updating as the values change. Reuse the TrailmarkCore HealthKit manager wherever it already fits.

Class 2.4 · Session 08 of 12

Sensing motion at the source — Core Motion

CoreMotionCMPedometerAccelerometerMotion activity
Learning outcomes
  • Access accelerometer, gyroscope, pedometer, and motion-activity data on the wrist, and explain why the wrist is the ideal place to sense motion.
  • Derive a simple signal — such as cadence or activity type — from raw motion data.
  • Recognize the energy cost of sensors, setting up the performance work in Course 3.
Integration day — no assignment. Motion sensing is wired into the watch app and we open the conversation about battery and sampling cost that Course 3 picks up.
03

Course Three

Intermediate watchOS — Live, Connected & Performant

The genuinely watch-hard topics: real-time workout sessions, wrist-to-pocket sync, power tuning, and complications.

Class 3.1 · Session 09 of 12

WatchConnectivity — making two apps one

WCSessionapplicationContexttransferFileuserInfo
Learning outcomes
  • Explain the WCSession transfer types — live messages, application context, userInfo, and file transfer — and when each fits.
  • Establish a session on both sides and sync a recorded activity from watch to phone.
  • Transfer a media file such as a voice memo across devices.
Assignment · build spec

"Pocket sync"

Record an activity and a voice memo on the watch, then sync both to the iPhone over WatchConnectivity so they appear in the iOS journey list. Choose the right transfer type for each payload and explain why.

Class 3.2 · Session 10 of 12

Live workout sessions & background running

HKWorkoutSessionHKLiveWorkoutBuilderBackground runtime
Learning outcomes
  • Start an HKWorkoutSession and stream metrics with a live workout builder.
  • Keep the app running and updating in the background during an activity.
  • Save the finished workout to HealthKit.
Assignment · build spec

"Go for a walk"

Build a live activity that starts on the watch, keeps updating heart rate, elapsed time, and energy while backgrounded, and saves a real HKWorkout on finish. Confirm it lands in the Health app.

Class 3.3 · Session 11 of 12

Performance & power for wearables

InstrumentsSampling ratesUI refreshSync batching
Learning outcomes
  • Profile CPU, energy, and memory on watchOS with Instruments.
  • Tune sensor sampling rates and UI refresh to cut battery cost.
  • Reduce connectivity overhead by choosing batched over chatty sync.
Assignment · build spec

"Make it last"

Profile one feature — for example live motion plus UI refresh plus sync. Make at least two optimizations and document the before/after measurements with a short note on the tradeoff you chose.

Class 3.4 · Session 12 of 12

Complications, Smart Stack & independence — capstone

WidgetKitComplicationsSmart StackIndependent app
Learning outcomes
  • Build a complication with WidgetKit and surface a key metric on the watch face and Smart Stack.
  • Make the watch app function independently where appropriate.
  • Integrate and present the complete Trailmark project end to end.
Capstone — no assignment. Final integration and presentation of the whole project across iPhone and Apple Watch.

Trailmark is a working name — rename it to fit your section. Assignments for classes 1–3 of each course are the in-class build expressed as acceptance criteria, so no extra work is created outside the session. Class 4 of each course is an integration / capstone day with no separate deliverable.
