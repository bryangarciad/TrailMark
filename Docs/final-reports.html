Final Reports

Each course ends by finishing and integrating the project — that's the final report. Three deliverables, one per course: the whole iOS app, then the whole watch app, then the complete cross-device project presented end to end. Same card format as the Assignments, scaled up to "is it a coherent product?"

What a final report is (and how it differs from an assignment) The weekly assignments (classes 1–3) each grade one feature. The final report (class 4) grades integration: do the pieces become one working, well-architected product? There is no new feature to invent — you connect what you already built, write it up, and demo it.
  • Written report (file upload, PDF): 1–2 pages — what you built, a short architecture note (how TrailmarkCore is shared), screenshots, and known limitations.
  • Demo recording (file upload, .mp4/.mov): 2–4 min walking through the integrated app end to end. Device recording for any sensor/watch features.
  • Source (website URL): repo link at the tagged commit (or a zipped project).
Recommended Canvas setup: Online submission with File Uploads + Website URL, 100 points. Course 3 adds a short live presentation.

Final Report · Course 1 100 pts iOS Session 1.4

"The iOS app, whole"

What it is. The integration day for Course 1. Connect the three previous builds — activity, media, and location — into one coherent iPhone app, anchored by a Journey detail screen that unites route, health, and media. The app should now be whole and ready for the wrist.

Deliverable requirements

  • A single iOS app with working Today (1.1), Field Journal (1.2), Recovery (1.3), and Journeys (1.4) features.
  • A Journey detail screen that shows the route polyline, geotagged memo pins, and the activity/health summary together.
  • Memos captured during a journey are geotagged and associated with it.
  • All platform logic (HealthKit, media, location) lives in TrailmarkCore — no framework imports in views.
  • Graceful states throughout (denied permissions, no data, empty lists) — no crashes.

The report should cover

  • An architecture note: what lives in the package vs the app, and why the split prevents duplication when the watch arrives.
  • Screenshots of each feature and the unified Journey screen.
  • Known limitations / what you'd do next.

Cumulative learning outcomes (Course 1)

  • Structure an app + shared Swift package, and justify the boundary.
  • Read and write HealthKit data with a correct authorization/privacy model.
  • Capture, persist, and play back media with sound metadata design.
  • Record a location track and present route + health + media as one journey.

Final Report · Course 2 100 pts watchOS Session 2.4

"TrailMark on the wrist"

What it is. The integration day for Course 2. The watch app is complete — home, memo, vitals, and motion — all built on the same TrailmarkCore as the phone. Motion sensing is wired in, and the battery/sampling conversation that Course 3 picks up is opened.

Deliverable requirements

  • A working watch app with Wrist Home (2.1), Wrist Memo (2.2), Live Vitals (2.3), and Motion (2.4).
  • Demonstrable code reuse: the watch imports the shared package with no duplicated model/manager code.
  • Wrist-appropriate design throughout — glanceable, focused, not a shrunk iPhone UI.
  • Live vitals and motion shown on a physical Apple Watch (sensors can't be simulated).

The report should cover

  • Which types/managers were reused unchanged from Course 1, and which views are watch-specific — quantify the reuse.
  • One design decision defended against a named watchOS guideline.
  • An early note on sensor/sampling cost (sets up Course 3.3).

Cumulative learning outcomes (Course 2)

  • Apply wearable design principles: glanceability, focused tasks, hierarchy, deferring to the phone.
  • Add a watch target and reuse a shared module instead of rewriting it.
  • Capture media and read live health metrics within wrist constraints.
  • Sense motion at the source with Core Motion and derive a useful signal.

Rubric

CriterionPts
All four watch features present and working (on device)35
Code reuse: no duplicated core; watch adds only views/glue25
Wrist-appropriate design & a justified guideline choice20
Written report (reuse analysis + design + sampling note)10
Demo recording + source complete10

Final Report · Course 3 · Capstone 100 pts iOS + watchOS Session 3.4

"The capstone — the whole project, end to end"

What it is. The program capstone. Integrate and present the complete TrailMark project across iPhone and Apple Watch: connectivity, live workouts, a tuned/performant build, and a complication on the watch face. Where appropriate the watch app stands on its own.

Deliverable requirements

  • Pocket sync (3.1): an activity and a memo recorded on the watch appear on the phone via WatchConnectivity.
  • Live workout (3.2): a session starts on the watch, updates while backgrounded, and saves a real HKWorkout to Health.
  • Performance (3.3): at least two measured optimizations carried over, with before/after numbers in the report.
  • Complication (3.4): a WidgetKit complication surfaces a key metric on the watch face / Smart Stack, backed by an App Group.
  • The watch app functions independently where appropriate.

The report + presentation should cover

  • A whole-system architecture diagram: the shared core, both apps, the complication, and the sync paths.
  • The transfer-type choice per payload (3.1) and the performance before/after (3.3).
  • A short live demo / presentation of the full project across both devices.

Cumulative learning outcomes (Course 3 + program)

  • Sync data and files between watch and phone with the right WatchConnectivity transfer type.
  • Run a live, background-capable workout session and persist the result.
  • Profile and tune a wearable feature for power, defending the tradeoffs with measurements.
  • Build a complication and present a coherent, integrated cross-device product.
