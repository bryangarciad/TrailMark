Course One · iOS — Foundations & the Shared Core

Course 1

1.1  Architecture & the HealthKit foundation
Swift PackageHealthKit authHKStatisticsQuery@Observable

Shared Swift package architecture

A Swift package is a self-contained module of code with its own Package.swift manifest. A local package lives inside your repo and is linked by one or more app targets. Putting models + managers in TrailmarkCore means the iPhone app and the watch app import the same types — fix a bug once, both platforms get it. The alternative (copy-pasting logic into each target) is where two-platform projects rot.

The rule we enforce all program long: views never import HealthKit/CoreLocation/etc. They read published state from a manager in the package. This keeps platform-specific frameworks out of the UI and makes the logic testable and reusable.

Key APIs: Package.swift · platforms · .library / .target · "Add Local…" package dependency.

HealthKit's authorization & privacy model

HealthKit gates every data type behind explicit user permission, split into read and write (share) sets. You request access with requestAuthorization(toShare:read:). The deliberate privacy twist: to avoid leaking what conditions a user might have, iOS never tells you whether READ access was granted — authorizationStatus(for:) only reports your write status. So you design for "the request finished," and treat a zeroed result as either "denied" or "genuinely no data." Never block the UI on a read-permission check.

Each requested type needs a purpose string in Info.plist (NSHealthShareUsageDescription / NSHealthUpdateUsageDescription) — without it the permission sheet silently never appears.

Watch out "The user denied read" and "there's no data yet" look identical to your code. Build one graceful empty state that covers both — don't show an error that implies a bug.

Key APIs: HKHealthStore · isHealthDataAvailable() · requestAuthorization(toShare:read:) · HKQuantityType(.stepCount).

Reading totals with HKStatisticsQuery

For "how many steps today" you don't want every individual sample — you want a sum. HKStatisticsQuery with the .cumulativeSum option asks HealthKit to do the aggregation for you over a date predicate. You read the result out in a specific unit (.count(), .meter(), .kilocalorie()). Quantities are always unit-typed — there is no "raw number," which prevents mixing meters and miles by accident.

let predicate = HKQuery.predicateForSamples(withStart: startOfDay, end: Date())
let q = HKStatisticsQuery(quantityType: HKQuantityType(.stepCount),
                          quantitySamplePredicate: predicate,
                          options: .cumulativeSum) { _, stats, _ in
    let steps = stats?.sumQuantity()?.doubleValue(for: .count()) ?? 0
}
store.execute(q)

Key APIs: HKStatisticsQuery · .cumulativeSum · HKQuery.predicateForSamples · HKQuantity.doubleValue(for:).

SwiftUI data flow with @Observable

Managers are marked @Observable (the Observation framework) and @MainActor. SwiftUI automatically re-renders a view when a property it actually read in body changes — no @Published, no ObservableObject boilerplate. HealthKit callbacks arrive on a background queue, so the manager hops back to the main actor before mutating published state.

Key APIs: @Observable · @MainActor · @State for ownership · .environment(_:) to inject · .task / .refreshable.

1.2  Capturing the moment — media management
AVAudioRecorderAVAudioSessionFileManagerAVAssetImageGenerator

Recording audio with AVFoundation

AVAudioRecorder writes microphone input straight to a file given a settings dictionary (format, sample rate, channels, quality). For voice memos, mono AAC in an .m4a container at 44.1 kHz is the sweet spot — small files, universal playback.

The audio session

Before recording or playing, you tell the system how your app uses audio via AVAudioSession: a category (.playAndRecord to record, .playback to play), a mode, and options like .duckOthers. You must setActive(true) to claim the session and deactivate when done so other apps' audio resumes. This is the single most common source of "my recording is silent / won't start" bugs.

Watch out Recording needs NSMicrophoneUsageDescription; the first record() triggers the mic permission prompt. No string → instant silent failure.

Persisting files & modeling metadata

The recorder writes to a temporary file; you move it into a durable location (we use Application Support/Media) and store metadata separately in a small JSON index. Critical detail: store the file's name, not an absolute URL. The app's container path changes between launches and across devices, so a saved absolute URL becomes a dangling pointer. Resolve name → live URL on demand.

Why this matters later Storing relative names is exactly what lets a memo recorded on the watch be transferred to the phone (Course 3.1) and still resolve correctly in the phone's container.

Video thumbnails & playback

AVAssetImageGenerator extracts a still frame from a video asset at a given time — your list thumbnail. Set appliesPreferredTrackTransform = true so portrait videos aren't rotated. Playback uses AVKit's VideoPlayer (video) or AVAudioPlayer (audio). Note AVAssetImageGenerator is iOS-only — guard it so the shared package still compiles for watchOS.

Key APIs: AVAudioRecorder · AVAudioSession · FileManager.url(for:in:) · AVAssetImageGenerator.image(at:) · AVPlayer / VideoPlayer.

1.3  Two-way HealthKit — workouts in, sleep & recovery out
HKWorkoutBuildersleepAnalysisHKStatisticsCollectionQuerySwift Charts

Writing a workout

Reading was half the story; now you write. The modern API is HKWorkoutBuilder (the old HKWorkout initializers are deprecated): beginCollection → add quantity samples (energy, distance) → endCollection → finishWorkout. Writing requires the share permission for those types. The payoff is verifiable: the workout shows up in Apple's Health app, proving your write path is real.

Sleep via category types

Not all health data is a number. Category types represent discrete states; sleepAnalysis samples carry a value like .asleepCore, .asleepDeep, .asleepREM, .inBed, .awake. "Last night's sleep" = sum the durations of the asleep stages within a sensible window (e.g. 6pm→noon) — bracketing the night correctly is the genuinely tricky part, since people sleep across midnight and nap.

Daily buckets with HKStatisticsCollectionQuery

For a 7-day trend you want one number per day. HKStatisticsCollectionQuery takes an anchor date and an interval (DateComponents(day: 1)) and returns a collection you enumerate to get per-bucket sums. It's the right tool any time you'd otherwise be tempted to run a query in a loop.

Visualizing with Swift Charts

Swift Charts turns an array of Identifiable points into a chart declaratively. A BarMark with x: .value("Day", day, unit: .day) and y: .value("kcal", energy) is the whole 7-day energy chart. Axis marks + .dateTime.weekday() label the days.

Key APIs: HKWorkoutBuilder · HKCategoryType(.sleepAnalysis) · HKCategoryValueSleepAnalysis · HKStatisticsCollectionQuery · Chart / BarMark.

1.4  Where it happened — location & the unified journey
CoreLocationCLLocationManagerMapKitMapPolyline

Location authorization & recording a track

CoreLocation has two authorization levels: When In Use (only while the app is foregrounded/active) and Always. We request when-in-use — enough to record a route while the user watches, and far less invasive. CLLocationManager is delegate-based: set desiredAccuracy and a distanceFilter (only report after moving N meters), call startUpdatingLocation(), and accumulate didUpdateLocations callbacks into a track.

First taste of the power tradeoff A bigger distanceFilter and coarser accuracy = fewer callbacks = less battery, at the cost of a blockier track. This is the exact knob Course 3.3 profiles.

Geotagging media

Because the LocationManager exposes the current coordinate, capturing a memo can stamp it with where it happened — just pass location.currentCoordinate into the media store. Coordinates are stored as plain lat/long doubles on the model (Codable-friendly), not as a CoreLocation type.

Drawing the route in MapKit

SwiftUI's Map takes a content builder. MapPolyline(coordinates:) draws the route line; Marker / Annotation drop memo pins. The "Journey detail" screen composes all three Course-1 builds — route + health + media — into one view, which is also the payload that later syncs to the watch.

Key APIs: CLLocationManager · requestWhenInUseAuthorization() · CLLocationManagerDelegate · Map · MapPolyline · Marker.

Course 2

2.1  Design for wearables + reusing the core
watchOS HIGGlanceabilityDigital CrownCode reuse

Wearable design principles

The watch is not a small phone. Apple's watchOS guidelines push four ideas: glanceability (the key info readable in ~2 seconds), focused tasks (one job per screen), clear hierarchy (one dominant element), and deferring to the phone for anything long or text-heavy. The Digital Crown is the precise scroll/adjust input — use it for values and long lists rather than tiny touch targets.

Adding a watch target & reusing TrailmarkCore

The watch app is a target in the same project, not a new project, created as a "Watch App for an Existing iOS App" so it's paired to the phone (WKCompanionAppBundleIdentifier). It links the same TrailmarkCore package, so models and managers are imported, not rewritten. The watch only adds new views and the occasional watch-specific manager method.

Watch out Translate, don't shrink. A glanceable headline metric (one big number) beats cramming the iPhone dashboard onto a 1.9″ screen.

Key APIs: watchOS target template · WKCompanionAppBundleIdentifier · shared package linking · SwiftUI on watchOS (List, NavigationStack).

2.2  Voice memos on the wrist — media under constraints
AVFoundation (watch)AVAudioSessionConstraints

Audio on watchOS

The same AVAudioRecorder / AVAudioPlayer and audio-session concepts from 1.2 work on the watch — which is the point: the AudioRecorder in TrailmarkCore is reused unchanged. What differs is the environment: limited storage, a smaller battery, no large keyboard, and a UI that must stay essential.

Designing within constraints

The wrist memo UI strips to: record/stop, a list, tap-to-play. No waveform scrubber, no rename flow, no multi-select — those defer to the phone. The lesson is reuse-with-restraint: same model and storage, deliberately different interface, and being able to articulate why each is appropriate to its device.

Key APIs: AVAudioRecorder · AVAudioSession (.playAndRecord) · shared MediaStore / MediaMemo.

2.3  HealthKit on the watch
HealthKit (watch)HKAnchoredObjectQueryLive updates

Sensor access: watch vs phone

The phone infers steps and energy; the watch is the source for heart rate and on-wrist motion. Authorization works the same way (requestAuthorization with its own Info.plist strings on the watch target), but the watch can deliver fresher, more frequent samples for vitals because the sensors are right there.

Live-updating queries

A one-shot statistics query is wrong for "current heart rate." HKAnchoredObjectQuery returns existing samples and keeps an updateHandler that fires as new samples arrive — you push each into published state and the SwiftUI view animates the change. (For continuous, high-frequency streaming during exercise you escalate to a workout session — Course 3.2.)

Watch out Heart rate cannot be simulated — the Simulator reports nothing. Demo and test live vitals on a real Apple Watch.

Key APIs: HKAnchoredObjectQuery · updateHandler · HKQuantityType(.heartRate) · unit count()/min · shared HealthKitManager.

2.4  Sensing motion at the source — Core Motion
CMPedometerCMMotionActivityManagerAccelerometerSampling cost

The Core Motion sensor family

Core Motion exposes the raw inertial sensors and higher-level processed signals. The accelerometer and gyroscope give motion/orientation; CMPedometer gives steps, distance, and cadence (steps/sec); CMMotionActivityManager classifies activity (walking / running / cycling / automotive / stationary) using a low-power coprocessor. The wrist is the ideal vantage point for all of this.

Deriving a signal from raw data

Raw numbers aren't insight. You derive: convert currentCadence (steps/sec) to steps/min, or reduce three accelerometer axes to a single magnitude (√(x²+y²+z²)). Each requires availability checks — not every device has every sensor (isStepCountingAvailable(), isActivityAvailable()).

The energy cost of sensors

Sensors aren't free. The accelerometer's default update rate can be 100 Hz; for a magnitude readout 10 Hz is plenty and far cheaper. Choosing the lowest rate that still answers your question is the core of wearable power discipline — and the setup for Course 3.3.

Key APIs: CMPedometer · CMMotionActivityManager · CMMotionManager.deviceMotionUpdateInterval · availability checks · CMMotionActivity.

3.1  WatchConnectivity — making two apps one
WCSessionapplicationContexttransferUserInfotransferFile

The session

WatchConnectivity links the paired iPhone and Watch apps. Both sides set a delegate and call WCSession.default.activate(). isReachable tells you whether the counterpart is awake and connected right now.

Choosing the right transfer type

The whole skill here is matching payload to mechanism:

  • sendMessage — live, immediate, needs the counterpart reachable now. Great for interactive request/response, wrong for a backgrounded phone.
  • updateApplicationContext — a single "latest state" dictionary that's coalesced: only the newest value survives. Perfect for mirroring today's summary to the wrist.
  • transferUserInfo — a FIFO queue of discrete dictionaries with guaranteed delivery, surviving relaunch. Right for "a workout finished."
  • transferFile — for binary payloads (a voice-memo file) with attached metadata.
Watch out A received file URL is only valid inside the delegate callback — copy the bytes out immediately or they're gone.

Key APIs: WCSession · WCSessionDelegate · updateApplicationContext · transferUserInfo · transferFile(_:metadata:).

3.2  Live workout sessions & background running
HKWorkoutSessionHKLiveWorkoutBuilderworkout-processing

Workout sessions (watchOS-only)

HKWorkoutSession is the privileged mode that lets a watch app keep running and collecting data during exercise. You pair it with an HKLiveWorkoutBuilder whose HKLiveWorkoutDataSource auto-collects the metrics relevant to the activity type. The builder's delegate fires as data arrives — you read statistics(for:) to update live heart rate, energy, and distance.

Background runtime

The reason a session matters: with the Workout Processing background mode enabled (a capability), the app keeps updating even when the wrist drops and the screen sleeps. Without it, a normal watch app is suspended almost immediately. On finish, endCollection → finishWorkout saves a real HKWorkout to Health.

Watch out Sessions, background runtime, and live metrics need a physical Apple Watch — the Simulator can start/save but won't stream sensor data.

Key APIs: HKWorkoutSession · HKLiveWorkoutBuilder · HKLiveWorkoutDataSource · HKWorkoutConfiguration · WKBackgroundModes workout-processing.

3.3  Performance & power for wearables
InstrumentsSampling ratesUI refreshBatching

Profiling with Instruments

You can't optimize what you don't measure. Instruments attaches to a running app and records traces: the Energy Log shows power impact, Time Profiler shows where CPU goes, Allocations shows memory. On a watch, energy is the budget that matters most.

The three big levers

  • Sensor sampling rate — the cheapest win. Halving an accelerometer/location update rate often halves that feature's energy with no felt difference (1.4's distanceFilter, 2.4's update interval).
  • UI refresh cadence — a timer ticking the display every 0.1s is wasteful if a 1s tick reads identically. Throttle redraws to what the eye needs.
  • Sync batching — chatty per-event messages cost a radio wake each time. Batch updates (applicationContext/transferUserInfo) instead of streaming live messages.
The discipline Every optimization is a tradeoff (freshness vs battery, smoothness vs cost). Document before/after numbers and name the tradeoff — measurement, not vibes.

Key APIs / tools: Instruments (Energy, Time Profiler, Allocations) · deviceMotionUpdateInterval · distanceFilter · TimelineView / timer cadence.

3.4  Complications, Smart Stack & independence
WidgetKitTimelineProviderApp GroupSmart Stack

Complications are WidgetKit widgets

Modern watch complications are built with WidgetKit, the same framework as iOS widgets. You declare a Widget with a configuration, a TimelineProvider, and a SwiftUI view that adapts to the complication families (accessoryCircular, accessoryRectangular, accessoryInline, accessoryCorner) on the watch face and in the Smart Stack.

Timelines & refresh

A complication doesn't run continuously; the system asks your TimelineProvider for entries and a refresh policy (e.g. .after(date)). Your app can also force a refresh with WidgetCenter.shared.reloadAllTimelines() when fresh data lands.

Sharing data via an App Group

The extension is a separate process from the app, so they can't share memory. An App Group gives both a shared UserDefaults suite / container — the app writes today's steps, the complication reads them. Add the same group id to the app and the extension.

Independence

Where it makes sense, the watch app should function without the phone nearby — reading its own HealthKit data, storing its own memos — which the shared-core architecture already enables. The capstone integrates everything end to end across iPhone and Apple Watch.

Key APIs: Widget · StaticConfiguration · TimelineProvider · widgetFamily · WidgetCenter.reloadAllTimelines() · App Groups / UserDefaults(suiteName:).
