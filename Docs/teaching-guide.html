Class-by-Class Build Guide

Exactly what to build in each session, which files to write, the concepts to land, and the demo checkpoint to end on — whether you're teaching it or working through it yourself. The in-class build is the assignment for classes 1–3 of each course.

How to read each session Build live = the files written/created in that session. Teach = the concepts to surface while coding (and to make sure you understand). Checkpoint = the visible result that proves it worked. File paths are relative to TrailmarkCore/Sources/TrailmarkCore/ (package) or the app target folders. Do Setup §1–3 before Course 1.1.

Course One · iOS — Foundations & the Shared Core

Course 1

Session 1.1  Architecture & the HealthKit foundation
Swift PackageHealthKitHKStatisticsQuery@Observable

Build live

  • Create the project + watch target + TrailmarkCore package skeleton (Setup §1–3).
  • Models/ActivitySummary.swift — plain value type, no HealthKit import; Support/Coding.swift.
  • Health/HealthKitManager.swift — requestAuthorization(), refreshToday(), and the sumQuantity(_:unit:since:) helper using HKStatisticsQuery.
  • App side: AppModel.swift, wire it into TrailMarkApp, build ContentView as a 4-tab TabView, then Views/Today/TodayDashboardView.swift + MetricCard.

Teach

  • Why a shared module prevents duplication later (the whole-program thesis).
  • HealthKit's authorization + privacy model; iOS never reveals whether read was granted — design for a zeroed/empty state, not an error.
  • Keep all HealthKit code in the package; the view only reads published state.

Checkpoint

Today tab shows steps/distance/energy from Health sample data, and a clean empty state when denied. → Assignment 1.1

Session 1.2  Capturing the moment — media management
AVFoundationAVAudioRecorderFileManagerThumbnails

Build live

  • Models/MediaMemo.swift — id, kind, relative file name, date, duration, optional coordinate.
  • Media/MediaStore.swift — save (move temp file in), JSON index, delete (also removes the file), iOS-only video thumbnail(for:).
  • Media/AudioRecorder.swift + AudioPlayer.swift.
  • App: Views/Journal/ — FieldJournalView (list + thumbnails), RecordAudioView, VideoCaptureView (UIImagePickerController), MemoDetailView (playback).

Teach

  • Capture sessions + permissions; configuring AVAudioSession.
  • Why store a relative file name, not an absolute URL (the container path changes between launches/devices).
  • Deleting metadata and the file together — no orphaned bytes.

Checkpoint

Record audio + video, see them listed with thumbnail + duration, play back in detail, delete removes the file. → Assignment 1.2

Session 1.3  Two-way HealthKit — workouts in, sleep & recovery out
HealthKit (write)HKWorkoutBuildersleepAnalysisSwift Charts

Build live

  • Models/WorkoutRecord.swift, Models/HealthInsights.swift (SleepSummary, EnergyTrendPoint, LiveVitals).
  • Extend HealthKitManager: save(_:activity:) via HKWorkoutBuilder; refreshLastNightSleep() (category query); refreshEnergyTrend() via HKStatisticsCollectionQuery.
  • App: Views/Recovery/RecoveryView.swift — sleep card, 7-day energy BarMark chart, "save sample workout" button.

Teach

  • Writing samples vs reading; verifying in the Health app builds trust in your code.
  • Category types & sleep stages; bracketing "last night" with a sensible date window.
  • Collection queries with a daily interval; turning stats into chartable points.

Checkpoint

Saved workout appears in Health; recovery screen shows last night's sleep + a 7-day energy chart. → Assignment 1.3

Session 1.4  Where it happened — location & the unified journey  Integration day · no assignment
CoreLocationMapKitMapPolylineGeotagging

Build live

  • Location/LocationManager.swift (when-in-use auth, record track, current coordinate), Models/RouteTrack.swift, Models/Journey.swift, Persistence/JourneyStore.swift.
  • Geotag memos at capture (pass location.currentCoordinate into MediaStore.add).
  • App: Views/Journeys/ — list, RecordJourneyView (live polyline), JourneyDetailView uniting route + stats + memo pins.

Teach

  • Connect the three prior builds into one screen; prepare the codebase to be consumed by a watch target.
  • The distanceFilter tradeoff — a first taste of the power conversation Course 3 finishes.

Checkpoint

Record a route (Freeway Drive on Sim), see polyline + geotagged memo pins on one Journey detail screen. The iOS app is now whole.

Course Two · watchOS — Designing & Building for the Wrist

Course 2

Session 2.1  Design for wearables + reusing the core
watchOS targetSwiftUI (watch)TrailmarkCore reuse

Build live

  • Confirm the watch target links TrailmarkCore (Setup §3b) — no duplicated model code.
  • WatchModel.swift (owns the same managers), watch TrailMarkCompanionApp, ContentView as a NavigationStack list.
  • Views/WristHomeView.swift — one glanceable headline metric + one quick action.

Teach

  • Wearable design principles: glanceability, focused tasks, hierarchy, Digital Crown ergonomics, when to defer to the phone.
  • Translating an iOS screen into a wrist-appropriate layout (one big number beats a dense dashboard).

Checkpoint

Glanceable watch home readable in ~2 seconds, importing the shared core. → Assignment 2.1

Session 2.2  Voice memos on the wrist — media under constraints
AVFoundation (watch)AVAudioSessionLocal storage

Build live

  • Views/WristMemoView.swift — reuse AudioRecorder + MediaStore unchanged; pared-down record/list/play UI.

Teach

  • Watch media constraints: storage, audio session, limited UI.
  • Reusing the media model while accepting a wrist-appropriate interface — contrast with the iOS capture experience and explain why they differ.

Checkpoint

Record, save (shared model), list, and play back a memo on the watch. → Assignment 2.2

Session 2.3  HealthKit on the watch
HealthKit (watch)HKAnchoredObjectQueryLive metrics

Build live

  • Extend HealthKitManager: startLiveVitals() / stopLiveVitals() with an anchored heart-rate query + today's steps/energy.
  • Views/LiveVitalsView.swift — live-updating heart rate, steps, energy.

Teach

  • How the watch's sensor access differs from the phone's.
  • Reusing the shared HealthKit layer wherever it already fits; only the live-streaming bit is new.
  • Honest note: real heart rate needs hardware — the Simulator shows 0.

Checkpoint

On device: heart rate / steps / energy update live on the wrist. → Assignment 2.3

Session 2.4  Sensing motion at the source — Core Motion  Integration day · no assignment
CoreMotionCMPedometerMotion activity

Build live

  • Motion/MotionManager.swift — pedometer cadence, motion-activity classifier, accelerometer magnitude.
  • Views/MotionView.swift — activity + derived signals.

Teach

  • Why the wrist is the ideal place to sense motion; deriving cadence/activity from raw data.
  • The energy cost of always-on sensors (sets up Course 3.3); note the 10 Hz vs 100 Hz choice.

Checkpoint

On device: activity label + cadence update as you move. Open the battery/sampling conversation.

Course Three · Intermediate watchOS — Live, Connected & Performant

Course 3

Session 3.1  WatchConnectivity — making two apps one
WCSessionapplicationContexttransferFiletransferUserInfo

Build live

  • Connectivity/ConnectivityManager.swift — one wrapper, both sides. applicationContext for latest summary, transferUserInfo for discrete records, transferFile for memos.
  • Wire onReceiveJourney/Workout/MediaFile in AppModel; saveAndSync + workout.onFinish in WatchModel.

Teach

  • Each transfer type and when it fits; why sendMessage is wrong for a backgrounded phone.
  • Files are only valid inside the delegate callback — copy out immediately.

Checkpoint

Record a memo/activity on the watch → it appears in the iOS list. → Assignment 3.1

Session 3.2  Live workout sessions & background running
HKWorkoutSessionHKLiveWorkoutBuilderBackground runtime

Build live

  • Workout/WorkoutSessionManager.swift (watchOS-only, #if os(watchOS)) — start session + live builder, stream HR/energy/distance, save on end, fire onFinish → sync.
  • Views/LiveWorkoutView.swift — live metrics + start/end.

Teach

  • Why HKWorkoutSession is watchOS-only and how the iOS side consumes only the result.
  • The Workout Processing background mode (Setup §5) is what keeps it alive while backgrounded.

Checkpoint

On device: start a walk, metrics update while backgrounded, finish saves a real HKWorkout to Health. → Assignment 3.2

Session 3.3  Performance & power for wearables
InstrumentsSampling ratesUI refreshSync batching

Build live (measure, then tune)

  • Profile a feature (live motion + UI refresh + sync) with Instruments (Energy, Time Profiler).
  • Tune: lower MotionManager update interval / LocationManager.distanceFilter; throttle the timer-driven UI ticks; prefer batched transferUserInfo/applicationContext over chatty messaging.

Teach

  • Reading CPU/energy/memory traces; turning a measurement into a decision.
  • Every optimization is a tradeoff — document before/after numbers, not vibes.

Checkpoint

Two measured optimizations with before/after and a note on the tradeoff. → Assignment 3.3

Session 3.4  Complications, Smart Stack & independence — capstone  Capstone · no assignment
WidgetKitComplicationsSmart StackApp Group

Build live

  • Create the Widget Extension target (Setup §6); drop in TrailMarkComplication.swift.
  • Publish steps to the App Group (WatchModel.publishStepsToComplication()); add the complication to a face.

Teach

  • WidgetKit complication families; sharing data across the app/extension boundary via an App Group.
  • Making the watch app function independently where appropriate; present the whole project end to end.

Checkpoint

Live steps on the watch face / Smart Stack. Final integration + presentation across iPhone and Apple Watch.
